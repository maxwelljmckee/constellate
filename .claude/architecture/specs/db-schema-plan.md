# SPEC — One-shot DB schema (MVP)

Status: **draft for review** — pre-implementation. Once approved, this becomes the authoritative reference for the single Drizzle migration that lands the full MVP data model.

Source documents: `architecture.md` §Data model, `todos.md` §3 (core tables, indexes, RLS), `specs/agents-and-scope.md`, `specs/onboarding.md` (seed protocol), `specs/research-task-prompt.md`, `todos.md` §17b (usage events).

This is the **one-shot migration** per build-plan rule: every table, every column, every index, every CHECK constraint, every RLS policy stub goes in one carefully-reviewed migration. No iterative schema evolution while building MVP slices.

## Identity model — Supabase Auth ownership boundary

**There is no `users` table in our `public` schema by design.** Identity is owned by Supabase Auth, which manages a `auth.users` table in a separate `auth` schema (the `auth` schema is provisioned by the Supabase platform and operated by its GoTrue service).

`auth.users` holds: `id` (uuid PK), `email`, `encrypted_password`, `email_confirmed_at`, raw OAuth identity rows, JWT claims metadata, etc. We **do not write to it directly** — Supabase Auth manages it via its API surface (sign-up, password reset, email confirmation, OAuth callback). Mutating it from our code risks breaking GoTrue's invariants.

**How our schema integrates with `auth.users`:**

1. Every domain table that's user-scoped has a `user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`.
2. RLS policies use `auth.uid()` (a Supabase-provided SQL function returning the JWT's subject UUID) to filter rows: `WHERE user_id = auth.uid()`.
3. The seed transaction (Slice 1, on signup webhook) inserts our domain rows (`agents`, `user_settings`, 20 `wiki_pages`) keyed off the `auth.users.id` Supabase already created.
4. User deletion (account-deletion flow, Slice 9) deletes the `auth.users` row; cascading FKs do the rest of the cleanup.

**If we ever need our own user-level data (display name, timezone, locale, signup metadata, profile image):**
- Tier-1: stash on `user_settings` if it's a single field or two
- Tier-2: introduce `public.user_profiles` as a sibling table when `user_settings` gets crowded
- **Never** add columns to `auth.users` directly — Supabase has a `raw_user_meta_data` jsonb on it for OAuth-provided fields, but that's GoTrue territory

**MVP user-level domain state we capture:**
- `user_settings` — `enabled_plugins`, `onboarding_complete`
- `agents` — per-user persona row(s)

That's it. Everything else (display name, etc.) comes from `auth.users` directly when needed.

## Enums

All as `pgEnum` (Postgres-native enum types). Listed once here; referenced by tables below.

```ts
wiki_scope_enum:    'user' | 'agent'
page_type_enum:     'person' | 'concept' | 'project' | 'place' | 'org' | 'source'
                  | 'event' | 'note' | 'profile' | 'todo' | 'agent'
edited_by_enum:     'ai' | 'user' | 'lint' | 'task'
agent_task_status:  'pending' | 'running' | 'succeeded' | 'failed' | 'cancelled'
agent_task_kind:    'research'
                  -- V1+ adds: 'podcast' | 'email_draft' | 'calendar_event' | 'brief'
wiki_log_kind:      'ingest' | 'agent_scope_ingest' | 'query' | 'lint' | 'task'
usage_event_kind:   'call_live' | 'ingestion_prefilter' | 'ingestion'
                  | 'agent_scope_ingestion' | 'plugin_research'
                  | 'tool_search_wiki' | 'tool_fetch_page'
artifact_kind_enum: 'research'
                  -- V1+ adds: 'podcast' | 'email_draft' | 'calendar_event' | 'brief'
call_type_enum:     'generic' | 'onboarding'
end_reason_enum:    'user_ended' | 'silence_timeout' | 'network_drop'
                  | 'app_backgrounded' | 'cancelled'
```

**Why pgEnum across the board:** Drizzle gets exhaustive type narrowing at the column type, server-side enforcement is automatic, and storage is minimal. Postgres makes additive value changes cheap (`ALTER TYPE ... ADD VALUE 'bar'` — non-blocking, no table lock, no rewrite). The expensive operations (remove / rename / reorder values) require recreating the type and rewriting the column — but those map to "we changed our mind about a category," which is rare in this codebase by design.

**Adding a new enum value** (e.g., a new plugin kind in V1+):
```sql
ALTER TYPE agent_task_kind ADD VALUE 'podcast';
```
That's the entire migration. Pair it with the TS schema regen (`drizzle-kit generate` picks up the change) and ship.

**Removing or renaming a value** (only do this if you must):
1. `CREATE TYPE foo_new AS ENUM (...)`
2. `ALTER TABLE ... ALTER COLUMN foo TYPE foo_new USING foo::text::foo_new` (rewrites the column — table lock proportional to row count)
3. `DROP TYPE foo`
4. `ALTER TYPE foo_new RENAME TO foo`

If we ever face this, it's a deliberate breaking change with a planned data migration — not something we'd hit by accident.

### 1. `agents`

Per-user persona records. MVP: exactly one row per user (the seeded `Assistant`).

| Column | Type | Constraints / Notes |
|---|---|---|
| `id` | `uuid` | PK, default `gen_random_uuid()` |
| `user_id` | `uuid` | FK → `auth.users.id`, ON DELETE CASCADE, NOT NULL |
| `slug` | `text` | NOT NULL |
| `name` | `text` | NOT NULL |
| `voice` | `text` | NOT NULL — Gemini Live voice id |

| `user_prompt_notes` | `text` | NULL — user-editable persona addendum |
| `root_page_id` | `uuid` | FK → `wiki_pages.id`, NULL until backfilled in seed transaction (no ON DELETE — application-level guard) |
| `is_default` | `boolean` | NOT NULL DEFAULT false |
| `created_at` | `timestamptz` | NOT NULL DEFAULT now() |
| `tombstoned_at` | `timestamptz` | NULL |

**Indexes:**
- `UNIQUE (user_id, slug)` — agent slug uniqueness within user
- `(user_id, is_default)` — fast lookup of default agent

**FK ordering note:** `agents.root_page_id` references `wiki_pages` and `wiki_pages.agent_id` references `agents`. Circular FKs handled by:
- `agents.root_page_id` defined as nullable
- Both FKs declared with `DEFERRABLE INITIALLY DEFERRED` so the seed transaction can insert both rows and resolve refs at commit

## RLS — MVP stubs

**Build-plan rule:** real RLS lands in Slice 9. For Slice 0 we:
1. Enable RLS on every public table.
2. Add a single permissive policy: `service_role` (the server's key) bypasses RLS via Postgres' built-in `bypassrls` attribute on the role itself — no policy needed for the service role.
3. Add **NO authenticated-role policies yet.** This means RxDB sync from the client will return zero rows until Slice 9 wires real per-table policies. That's acceptable since the client doesn't exist yet (Slice 0c).

```sql
ALTER TABLE wiki_pages ENABLE ROW LEVEL SECURITY;
ALTER TABLE wiki_sections ENABLE ROW LEVEL SECURITY;
-- ... and so on for every table
```

**Why this approach:** prevents accidental leakage during dev (no permissive default), forces us to write deliberate policies in Slice 9 against a fully-populated dev DB.

## Migration generation flow

1. Write all schema files in `apps/server/src/db/schema/`.
2. Run `pnpm db:generate` → drizzle-kit emits `drizzle/0000_*.sql`.
3. **Hand-edit the generated SQL** to add things Drizzle can't express:
   - `CREATE EXTENSION` statements at the top
   - Circular FK `DEFERRABLE INITIALLY DEFERRED` clauses
   - CHECK constraints with multi-column predicates
   - Partial indexes with `WHERE` clauses (Drizzle 0.28 supports some, but verify generated output)
   - GIN indexes with `jsonb_path_ops` op class
   - tsvector FTS index
   - `set_updated_at()` function + triggers
   - `ENABLE ROW LEVEL SECURITY` per table
4. Apply via `pnpm db:migrate` against the cloud Supabase.
5. Sanity-check via `psql` or Drizzle Studio — confirm row count, FK list, index list.
6. Generate types: `drizzle-kit generate` outputs TS types alongside the schema; we re-export from `packages/shared/src/db.ts` for cross-app consumption.

## Open questions for review

1. **`wiki_log.ref` as jsonb or as separate FK columns?** Chose jsonb for flexibility (kinds vary in what they reference). Tradeoff: no FK enforcement on the ref. Acceptable since `wiki_log` is append-only audit.
2. **`agent_tasks.result_artifact_id` as uuid + kind, no FK.** Per todos.md §3 — polymorphic artifact ref. Resolved at read time. Confirmed in spec.
3. **`call_transcripts.session_id` as text or uuid?** Gemini Live's session id is opaque; using text for safety. Could narrow to uuid if Google guarantees the format.
4. **`tags.color` as text (hex string) or normalized enum?** Text for flexibility — UI may want palette additions without schema churn.
5. **Should `wiki_pages.agent_id` ON DELETE be CASCADE or RESTRICT?** Per spec, default Assistant is non-deletable at MVP. RESTRICT prevents accidents; for V1+ custom agent deletion, application logic does cascade-tombstone, not DB cascade.
6. **Drizzle vs hand-rolled SQL split.** Some constraints (deferrable FKs, partial indexes, GIN op classes) need hand-edits to the generated `0000_*.sql`. That's fine for the one-shot, but means `drizzle-kit generate` won't be a round-trip-safe operation later if we add tables. Slice-by-slice we'll add per-feature migrations on top, never regenerating 0000.
